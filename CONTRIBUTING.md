**GUIDELINES FOR CONTRIBUTING**
*To keep the curriculum high-quality and professional, please follow these steps:*

---

**1. CORE PRINCIPLES**
* **Correctness First:** Ensure all terminal commands are accurate for the year **2026**.
* **Clarity:** Use simple language for complex topics like **Kernel Compilation**.
* **Standardization:** Follow the existing Markdown style using `**` and `*`.
* **No Bloat:** Keep explanations concise. We want a roadmap, not an encyclopedia.

---

**2. HOW TO CONTRIBUTE**
* **Step 1: Fork the Repo:** Create your own copy of the project to work on.
* **Step 2: Create a Branch:** Use a descriptive name like `feature/add-docker-guide`.
* **Step 3: Commit Changes:** Write clear, professional commit messages.
* **Step 4: Push & Pull Request:** Submit your PR for review.

---

**3. WHAT WE NEED**
* **Phase Additions:** Deep-dives into tools like **Sway**, **SELinux**, or **eBPF**.
* **Technical Fixes:** Updating commands that may have changed in newer releases.
* **Code Snippets:** Adding clean, functional **Bash** scripts or **Kernel** configs.
* **Resources:** Linking to high-quality documentation or the **Arch Wiki**.

---

**4. PULL REQUEST CHECKLIST**
* [ ] Does the code/command work on a fresh Linux install?
* [ ] Is the Markdown formatting consistent with the main `README.md`?
* [ ] Did you avoid "robot logic" or generic AI filler?
* [ ] Are the explanations functional and logical?

---

**COMMUNITY CODE OF CONDUCT**
* Respect all skill levels. We were all `cd /` beginners once.
* Provide constructive feedback on Pull Requests.
* Stay focused on **Linux**, **DevOps**, and **Kernel** development.

---

**THANK YOU FOR CONTRIBUTING!**
*Your help makes this roadmap better for every aspiring Linux engineer.*
